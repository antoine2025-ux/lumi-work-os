### 1.2 Root Redirect Logic

`src/app/(dashboard)/w/[workspaceSlug]/org/page.tsx`:

```
getOrgPermissionContext()
  → null → OrgEmptyState ("Create workspace")
  → OWNER or ADMIN → redirect /w/${slug}/org/admin
  → has OrgTeam.leaderId === userId → redirect /w/${slug}/org/my-team
  → else → redirect /w/${slug}/org/profile
```

**Issue:** Both MEMBER and VIEWER land on `/profile`. There is no distinction between a plain MEMBER and a VIEWER at the root redirect.

### 1.3 Layout Behavior

- Checks `isOrgCenterForceDisabled()` → shows static disabled message if true

- Calls `ensureOrgContextSynced(workspaceId)` — non-blocking background Loopbrain sync
- Wraps children in `<OrgPermissionsProvider>` + `<OrgLayoutClient>`
- `showHeader={false}` — relies on parent dashboard layout header

## 2. My Profile — Complete Field Audit

### 2.1 Page Chain

```
MyProfilePage (server component)
  → getOrgPermissionContext()  [auth]
  → prisma.user.findUnique()   [name, email]
  → prisma.orgPosition.findMany() [title, teamId, startDate, employmentType, location, timezone]
  → prisma.capacityContract.findFirst()  [weeklyCapacityHours]
  → getUserWorkload(userId, workspaceId)  [totalCapacity, allocatedHours, availableHours, projects]
  → getUserTimeOff(userId, workspaceId)  [upcomingAbsences, pendingRequests, PTO balance]
  → prisma.workspaceMember.findFirst()  [role]
  → prisma.personManagerLink.findMany()  [manager links]
  → prisma.orgTeam.findMany() where leaderId = userId  [led teams]
  → prisma.leaveRequest.findMany() [pending approvals — conditional on manager/admin]
  → prisma.wikiPage.findMany()  [last 5 pages + count]

Renders:
  BasicInfoSection
  EmploymentDetailsSection
  CurrentWorkloadSection
  TimeOffSectionWrapper → TimeOffSection + RequestTimeOffDialog
  PendingActionsSection (managers/admins only)
  WikiContributionsSection
  Team Membership card (inline JSX)
  Capacity & Availability card (inline JSX)
  My Responsibilities card (inline JSX)
```

### 2.2 Section-by-Section Breakdown

#### Section 1: Basic Information (`BasicInfoSection`)
| Field | Data Source | Wired? | Editable? | Empty State |
|-------|-------------|--------|-----------|-------------|
| Avatar | First letter of name | Yes (derived) | No | Shows "?" |
| Full name | `user.name \|\| onboardingState.adminName \|\| "User"` | Yes | Yes (EditBasicInfoDialog) | Falls back to "User" |
| Role/Title | `primaryPosition.title \|\| onboardingState.adminRole \|\| "Team Member"` | Yes | Yes (EditBasicInfoDialog) | Falls back to "Team Member" |
| Email | `user.email` | Yes | No (shown as disabled in dialog) | Hidden if null |

**Edit:** `EditBasicInfoDialog` calls:
- `PUT /api/org/people/[positionId]/name` — updates user's name
- `PUT /api/org/people/[positionId]/title` — updates position title

**Bug/Concern:** The dialog is called with `positionId` (OrgPosition ID), not `userId`. The API route path parameter is named `[personId]` but receives a position ID. This is a naming mismatch that could confuse future maintainers.

**Edit button visibility:** Only shown when `positionId` exists (i.e., user has an active position). Users without a position cannot edit their name.

#### Section 2: Employment Details (`EmploymentDetailsSection`)

**Field-level permissions:** `startDate` and `employmentType` are admin-only. `location` and `timezone` are self-editable.

**Edit button visibility:** Only shown when all of `positionId`, `userId`, `workspaceId` are present, and `canEdit=true`.

**Relations:** parent/children (self-referential hierarchy), team, user, workspace, jobDescription, roleCard, projectMembers, projectAssignees, projectPersonLinks.

**Indexes:** workspaceId, workspaceId+isActive, workspaceId+userId, workspaceId+archivedAt, workspaceId+mergedIntoId, teamId, parentId, level, jobDescriptionId, workspaceId+teamId+userId.

### 7.2 Key Related Models

**PersonManagerLink:** Explicit manager relationships (workspaceId, managerId, personId, startAt, endAt, linkType). Used alongside OrgPosition.parentId for manager inference.

**CapacityContract:** (workspaceId, personId, weeklyCapacityHours, effectiveFrom, effectiveTo). One person can have multiple contracts with time windows.

**LeaveRequest:** (workspaceId, personId, startDate, endDate, leaveType, status=PENDING/APPROVED/DENIED, daysCount). Managed by managers/admins via PendingActionsSection.

**OrgDepartment:** (workspaceId, name, isActive). No direct leaderId field — leadership inferred from highest-level OrgPosition within dept.

**OrgTeam:** (workspaceId, name, departmentId, leaderId, isActive). Has explicit leaderId (User FK).

**JobDescription:** Shared template model (workspaceId, title, summary, level, jobFamily, responsibilities[], requiredSkills[], preferredSkills[], keyMetrics[]). Linked to OrgPosition via jobDescriptionId.

**RoleCard:** Per-position role definition (workspaceId, positionId?, roleName, jobFamily, level, roleDescription, responsibilities, keyMetrics). Has skillRefs relation.

### 7.3 My Profile Data Path (Full Trace)

```
Browser → GET /w/[slug]/org/profile
  → WorkspaceOrgLayout (layout.tsx):

      ensureOrgContextSynced() [non-blocking background]
  → MyProfilePage (profile/page.tsx):
      getOrgPermissionContext() [second call — React.cache deduplicates]
      Promise.all([
        prisma.user.findUnique({ id: userId, select: {name, email} })
        prisma.orgPosition.findMany({
          where: { workspaceId, userId, isActive: true, archivedAt: null },
          include: { team: { include: { department: true } } }
        })
        prisma.capacityContract.findFirst({
          where: { workspaceId, personId: userId },
          orderBy: { effectiveFrom: 'desc' }
        })
        prisma.workspaceOnboardingState.findUnique({ workspaceId })
        getUserWorkload(userId, workspaceId)
        getUserTimeOff(userId, workspaceId)
        prisma.workspaceMember.findFirst({ userId, workspaceId, select: {role} })
        prisma.personManagerLink.findMany({ managerId: userId, workspaceId })
        prisma.orgTeam.findMany({ leaderId: userId, workspaceId, include: positions })
      ])
      [conditional] prisma.leaveRequest.findMany() [if isAdmin || isManager]
      [conditional] prisma.user.findMany() [requestor names]
      Promise.all([
        prisma.wikiPage.findMany({ workspaceId, createdById: userId, take: 5 })
        prisma.wikiPage.count({ workspaceId, createdById: userId })
      ])
  → BasicInfoSection receives: displayName, displayRole, email, positionId

  → TimeOffSectionWrapper receives: { upcomingAbsences, pendingRequests, remainingPTO, usedPTO, totalPTO, onRequestTimeOff }
  → PendingActionsSection receives: requests[] (with person info embedded)
```
