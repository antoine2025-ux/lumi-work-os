## Section 2: Current State Snapshot

### A. Storage Model (MOST IMPORTANT)

**Content Format:**
- **Storage:** HTML string in `WikiPage.content` field (String type)
- **Location:** PostgreSQL `wiki_pages.content` column
- **Format:** Raw HTML generated by `contentEditable` + `document.execCommand`

**Title Storage:**
- **Separate field:** `WikiPage.title` (String)
- **Slug:** `WikiPage.slug` (String, unique per workspace via `[workspaceId, slug]`)

**Metadata Fields (separate from content):**
- `excerpt`: String? (auto-generated, first 200 chars)
- `parentId`: String? (hierarchical parent)
- `order`: Int (for sorting, default 0)
- `isPublished`: Boolean (default true)
- `tags`: String[] (array)
- `category`: String (default 'general')
- `permissionLevel`: String (default 'team', can be 'personal')
- `workspace_type`: String? (values: 'personal', 'team', or custom workspace IDs)
- `view_count`: Int? (default 0)
- `is_featured`: Boolean?
- `last_viewed_at`: DateTime?
- `ai_analysis`: Json? (unstructured AI metadata)
- `quality_score`: Decimal? (3,2 precision)
- `createdAt`, `updatedAt`: DateTime
- `createdById`: String (FK to User)

**Versioning:**
- **Model:** `WikiVersion` table
- **Fields:** `id`, `pageId`, `content` (String), `version` (Int), `createdAt`, `createdById`
- **Implementation:** Auto-created on PUT when content changes (`src/app/api/wiki/pages/[id]/route.ts:285-296`)
- **Version numbering:** Incremental (nextVersion = max + 1)

**Autosave/Drafts:**
- **NO autosave** - manual save only (`handleSave` in page detail)
- **Draft state:** Uses `sessionStorage` for AI-generated drafts (`pendingPageDraft` key)
- **Published state:** `isPublished` boolean field (default true)

### B. Editor Architecture

**Editor Library:**
- **Current:** `contentEditable` div + `document.execCommand` (legacy browser API)
- **TipTap installed but NOT used:** `@tiptap/react`, `@tiptap/starter-kit` in `package.json` but no imports found
- **No modern editor:** Not using TipTap, ProseMirror, Lexical, Slate, or Quill

**Block Structure:**
- **NOT block-based** - single rich text field
- **NO block IDs** - content is flat HTML string
- **NO reordering** - no drag-and-drop for blocks

**Formatting Representation:**
- HTML tags: `<b>`, `<i>`, `<u>`, `<h1-6>`, `<ul>`, `<ol>`, `<blockquote>`, `<pre><code>`, `<table>`
- Inline styles for alignment (left/center/right)
- Embed placeholders: `<div class="embed-placeholder" data-embed-id="...">` (HTML string)

**Editor Features:**
- Bold, italic, underline, strikethrough
- Headings (H1-H3)
- Lists (ordered, unordered)
- Quotes, code blocks
- Tables (insert/delete)
- Links, images (via prompts)
- Undo/redo (browser native)
- Clear formatting
- Slash command detection (basic, triggers embed palette)

## Section 4: Migration Feasibility

### Storage Format Change (HTML → Block JSON)

**Effort:** **HIGH**

**What would need to change:**
1. **Database schema:**
   - Add `content_blocks` JSON field (or new `WikiBlock` table)
   - Migration script: HTML → block JSON converter
   - Keep `content` field for backward compatibility (or deprecate)

2. **Editor rewrite:**
   - Replace `contentEditable` with TipTap/Lexical
   - Implement block renderer
   - Block serialization/deserialization

3. **Version history:**
   - Migrate `WikiVersion.content` (HTML) to block format
   - Or keep versions as-is and convert on-demand

4. **API changes:**
   - Update `POST /api/wiki/pages` to accept block JSON
   - Update `PUT /api/wiki/pages/[id]` to handle blocks
   - Update `GET` endpoints to return block format

5. **Frontend components:**
   - Replace `RichTextEditor` with block-based editor
   - Update `WikiPageBody` renderer
   - Update embed system (currently HTML strings)

6. **Search/indexing:**
   - Update `WikiChunk` extraction to work with blocks
   - Re-index all existing pages

**Risk mitigation:**
- Dual-write period (save both HTML and blocks)
- Gradual migration (new pages = blocks, old pages = HTML until converted)
- Rollback plan (keep HTML as source of truth initially)

## Summary

**Current State:** Basic HTML-based editor with manual save, no block system, no templates, no import/export. Foundation exists (versioning, hierarchy, spaces) but needs significant upgrades for Notion/Slite parity.

**Biggest Blocker:** Storage format (HTML string) prevents block-based features without major migration.

**Quick Wins:** Autosave, templates, mentions (if using TipTap).

**Strategic Decision Needed:** Migrate to block-based storage now, or build features on HTML and migrate later (higher technical debt).
